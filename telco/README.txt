telco benchmark — data files
=============================

Source: Mike Cowlishaw's "Telco" benchmark
        https://speleotrove.com/decimal/telco.html
        (specification: https://speleotrove.com/decimal/telcoSpec.html)

This directory holds the telco test inputs in the three published encodings, at
two sizes, plus the reference outputs used as correctness gates. All input files
were downloaded verbatim from the URL above (the 1e6 files came zipped and were
inflated; the zips are kept for provenance).

Each of the three encodings represents the SAME sequence of call durations, so
for a given size all three encodings produce the SAME output; there is therefore
one reference-output file per size, not per encoding.

PRIMARY tests — one million numbers (used for performance runs)
------------------------------------------------------------------------------
  expon180.1e6         packed decimal (BCD)         8,000,000 bytes = 1e6 * 8
  expon180.1e6b        binary  (big-endian uint64)  8,000,000 bytes  <-- read by our programs
  expon180.1e6r        decimal64                     8,000,000 bytes
  expon180.1e6.expected   reference output: 1,000,000 lines, CRLF-terminated
  (expon180-1e6.zip, expon180-1e6b.zip, expon180-1e6r.zip — original archives)

  Exponential distribution, mean 180 (actual mean 180.2, max value 2641).
  These files are exactly 8,000,000 bytes (no trailing byte).

  Control totals (from the specification, telcoSpec.html) that a correct run must
  print for the one-million-number input:

       sumT (total, incl. tax) = 1004737.58
       sumB (basic tax)        =   57628.30
       sumD (distance tax)     =   25042.17

  expon180.1e6.expected is NOT published as a file upstream. It was GENERATED by
  running the benchmark over expon180.1e6b and VALIDATED two independent ways:
    (1) the printed sumT/sumB/sumD match the spec control totals above, and
    (2) all seven ports (C, Java, Python, Rust, Go, Swift, C#) produce
        byte-identical output.

SMOKE tests — ten numbers (worked example; quick correctness check)
------------------------------------------------------------------------------
  telco.test           packed decimal (BCD)          81 bytes
  telco.testb          binary  (big-endian uint64)   81 bytes
  telco.testr          decimal64                      81 bytes
  telco.expected       reference output: 10 lines, CRLF-terminated

  Durations: 308, 76, 68, 323, 82, 329, 14, 101, 522, 50.
  Each 10-number file is 81 bytes = 10*8 data bytes + one trailing 0x0a byte;
  the trailing byte is not a complete 8-byte record and is ignored by the read
  loop (which advances only on a full-record read). Files kept byte-for-byte as
  published.

  telco.expected was constructed from the worked example on telco.html and
  validated against its published 10-number control totals (sumT=8.91,
  sumB=0.50, sumD=0.22) AND confirmed byte-identical to a port run on telco.testb.

Correctness gate
------------------------------------------------------------------------------
  A run is correct iff its produced output matches the corresponding *.expected
  file byte-for-byte (and the printed sumT/sumB/sumD match the totals above).
  Reference outputs use CRLF line endings, matching the programs' "%s\r\n" writes.

Computation (from the benchmark specification)
------------------------------------------------------------------------------
  For each call, n = duration:
    rate  = 0.0013  if n is even (basic/"L" call), else 0.00894 (distance/"D")
    price = quantize(rate * n, 0.01, round-half-even)            ; banker's round
    btax  = quantize(price * 0.0675, 0.01, round-toward-zero)    ; truncated
    dtax  = quantize(price * 0.0341, 0.01, round-toward-zero)    ; D calls only
    total = price + btax (+ dtax for D calls)                    ; -> output line
    sumT += total ; sumB += btax ; sumD += dtax
